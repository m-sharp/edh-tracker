## Spacing Scale

Unchanged from Phase 2. MUI default `theme.spacing` unit is 8px.

| Token | Value | MUI spacing() call | Usage |
|-------|-------|--------------------|-------|
| xs | 4px | `theme.spacing(0.5)` | Icon gaps, inline padding within chips |
| sm | 8px | `theme.spacing(1)` | Compact element spacing, gap between paired icon buttons |
| md | 16px | `theme.spacing(2)` | Default element spacing, form field separation |
| lg | 24px | `theme.spacing(3)` | Section padding, vertical rhythm between content blocks |
| xl | 32px | `theme.spacing(4)` | Layout gaps between major sections |
| 2xl | 48px | `theme.spacing(6)` | Major section breaks |
| 3xl | 64px | `theme.spacing(8)` | Page-level spacing (e.g., top margin below AppBar) |

Exceptions (unchanged from Phase 2):
- AppBar height: 64px (MUI Toolbar default) — not overridden
- Touch targets: minimum 44px height for all interactive elements on mobile — `size="medium"` buttons satisfy this; `size="small"` must have `sx={{ minHeight: 44 }}` in touch contexts

Phase 5 additions:
- Player stat cards in PlayersTab: `padding: theme.spacing(2)` (md=16px) inside each card; `gap: theme.spacing(2)` between cards
- "Add Deck" button in PlayerDecksTab: `mt: theme.spacing(2)` (md=16px) above the DataGrid, aligned right

**Source:** Phase 2 UI-SPEC spacing scale carried forward. Phase 5 card/button spacing derived from established md/sm conventions.

## Typography

Unchanged from Phase 2. All roles carry forward exactly.

| Role | Size | Weight | Line Height | MUI variant | Usage |
|------|------|--------|-------------|-------------|-------|
| Display | 28px | 700 (bold) | 1.2 | h4 | Pod names, page-level headings |
| Heading | 20px | 700 (bold) | 1.2 | h6 | Section headings, player card names |
| Body | 16px | 400 (regular) | 1.5 | body1 | Primary readable content, stat row text |
| Label | 14px | 400 (regular) | 1.4 | body2 | Secondary info, DataGrid cell text, chip labels, stat numbers within cards |

Two weights only: 400 (regular) and 700 (bold). No intermediate weight.

Phase 5 application:
- Player card name: `variant="body1"` (16px, weight 400) rendered as a `Link` to `/player/{id}`
- Player card stat row (record + points + kills): `variant="body2"` (14px, weight 400, color `text.secondary`)
- Player role chip label: `variant="body2"` (14px) inside MUI `Chip size="small"`
- /deck/new form labels: `variant="body1"` (16px) for field labels; field input text inherits MUI TextField defaults (Roboto 16px)
- /deck/new page heading: `variant="h4"` (28px, Josefin Sans, 700) — "New Deck"

**Source:** Phase 2 UI-SPEC typography contract carried forward. Phase 5 role assignments are direct application of existing scale.

## Color

Unchanged from Phase 2 dark theme palette.

| Role | Value | MUI palette key | Usage |
|------|-------|-----------------|-------|
| Dominant (60%) | `#0f1117` | `palette.background.default` | App background, body fill, /deck/new page background |
| Secondary (30%) | `#1a1a2e` | `palette.background.paper` | Cards, AppBar, elevated surfaces, player stat cards |
| Accent (10%) | `#c9a227` | `palette.primary.main` | See accent reserved list below |
| Accent hover | `#e0b83a` | `palette.primary.light` | Hover state for accent elements |
| Primary text | `#e8edf5` | `palette.text.primary` | All primary readable text |
| Secondary text | MUI dark-mode default opacity on `#e8edf5` | `palette.text.secondary` | Stat rows, secondary labels |
| Destructive | MUI `error.main` (`#f44336`) | `palette.error.main` | Remove Player action only |

**Accent reserved for (this phase):**
1. Primary `variant="contained"` buttons — "Create Pod" submit, "Save" on /deck/new
2. Active Tab indicator underline (existing, unchanged)
3. Focused TextField outline on /deck/new and Create Pod modal
4. AppBar pod-switcher dropdown "Create new pod" MenuItem text label — accent color to distinguish it from pod list items

Accent is NOT used for: player card borders, stat text, role chips, the AppBar Select outline by default, icon buttons for Promote/Remove.

**Player card elevation:** Use `Paper` component with `elevation={2}` and `sx={{ bgcolor: "background.paper" }}`. This renders at `#1a1a2e` (secondary surface), clearly separated from the `#0f1117` page background. No custom border needed — elevation provides the visual separation.

**Source:** Phase 2 UI-SPEC color contract carried forward. Phase 5 additions: player card surface, "Create new pod" accent label — both derived from existing palette.

## Visual Hierarchy

### HomeView Empty State (POD-02)

Onboarding screen for users with no pods. Focal point sequence:
1. Centered heading: "Welcome to EDH Tracker" — `variant="h4"` (Display, 28px Josefin Sans)
2. Supporting body copy — `variant="body1"` centered below heading
3. "Create a Pod" button — `variant="contained"` (accent gold), centered below copy

No secondary CTA. Joining via invite link is handled through the /join route; the HomeView empty state focuses on creation only (D-07).

### AppBar Pod Switcher (POD-01, D-03)

The existing `PodSelector` MUI `Select` is extended to add a "Create new pod" MenuItem at the bottom of the list, visually separated from pod links by a `Divider`. The "Create new pod" text renders in accent gold (`color: "primary.main"`) to distinguish it as an action, not a navigation destination. All other MenuItem text renders in `text.primary`.

The playing cards icon (`SvgIconPlayingCards`) becomes a `Link` to `/` using `component={Link}` (D-33), matching the established Phase 3/4 pattern. No visual change — icon style unchanged; only the interactivity is added.

### Pod Players Tab Card Layout (POD-04, D-20)

Each player card is a `Paper` component with `elevation={2}` using a flex layout. Internal layout:

```
[ player name (Link) ] ............. [ Manager chip ]    [ icon buttons — manager only ]
[ stat row: 5W/3/2L • 42 pts • 12 kills ]
```

- Player name: `body1`, link color inherits MUI link default (accent gold on dark theme)
- Role chip: MUI `Chip size="small"` — only shown for managers (`label="Manager"`); members show no chip (cleaner layout, avoids "Member" noise)
- Stat row: `body2`, `color="text.secondary"`, rendered as a single line `"5W / 3L / 2L • 42 pts • 12 kills"` — separator is `" • "` (middle dot with spaces)
- Promote icon: `PersonAddIcon`, rendered as `TooltipIconButton` with `title="Promote to Manager"` — member-only, manager-only visibility
- Remove icon: `PersonOffIcon`, rendered as `TooltipIconButton` with `title="Remove from pod"`, `color="error"` — manager-only, not shown for caller's own card

Skeleton loading: while pod-scoped stats are fetching (if two-request approach used), the stat row renders as `Skeleton variant="text" width={200}` in place of the stats line. Player name and role chip render immediately from the first request.

### Pod Decks Tab Sort (POD-03)

DataGrid `initialState.sorting` set to `[{ field: "record", sort: "desc" }]`. The existing `RecordComparator` on the Record column definition handles sort logic. No visual changes beyond the default sort order.

### /deck/new Route (DECK-01)

Full-page route at `/deck/new`. Layout matches the existing NewGameView stacked card pattern from Phase 4 — centered `Container maxWidth="sm"`, `Paper elevation={2}` card, form content inside.

Page heading: `Typography variant="h4"` — "New Deck"
Subheading (optional, omit if not needed): none

Form fields in order:
1. Name (required) — `TextField fullWidth`
2. Format (required) — MUI `Autocomplete` or `Select` with format options
3. Commander (required, Commander format only) — `Autocomplete freeSolo` — conditionally shown
4. Partner Commander (optional, Commander format only) — `Autocomplete freeSolo` — conditionally shown

Commander/Partner fields use `freeSolo={true}` with a "Create {input}" option rendered as the last item when no match is found in the list. Typing a name not in the list and pressing Enter (or clicking "Create X") triggers `PostCommander(name)` and uses the returned ID for the deck create request.

Action buttons row below form (right-aligned on desktop, stacked on mobile xs):
- "Cancel" — `variant="outlined"` — navigates to `/player/{callerId}`
- "Save" — `variant="contained"` (accent gold) — submits form

**Source:** D-09 through D-19 (CONTEXT.md). Form layout pattern from Phase 4 NewGameView (stacked card, Container maxWidth="sm").

## Interaction States

### Create Pod Modal

| State | Behavior |
|-------|----------|
| Pod name empty | "Save" button disabled |
| Pod name filled | "Save" button enabled (accent gold) |
| Submitting | "Save" button shows `CircularProgress size={20}` inside button; input disabled |
| Success | Modal closes; auto-navigate to `/pod/{newPodId}` |
| Error | Error copy shown inline below button; modal stays open; input re-enabled |

### /deck/new Form

| State | Behavior |
|-------|----------|
| Format not selected | Commander/Partner fields hidden |
| Format = "Commander" selected | Commander field appears (required); Partner field appears (optional) |
| Format != "Commander" | Commander/Partner fields hidden |
| Commander field — typing known value | Autocomplete dropdown shows matching options |
| Commander field — typing unknown value | "Create {input}" appears as last option in dropdown |
| All required fields filled | "Save" enabled |
| Missing required field | "Save" disabled |
| Submitting | "Save" shows `CircularProgress`; all fields disabled |
| Success | Navigate to `/player/{callerId}/deck/{newDeckId}` |
| Error | Error copy shown inline; form re-enabled |

### AppBar Pod Switcher

| State | Behavior |
|-------|----------|
| User has no pods | "No pods" disabled MenuItem shown; "Create new pod" shown below Divider |
| User has pods | Pod names listed as MenuItems (navigate on select); "Create new pod" shown below Divider in accent gold |
| "Create new pod" selected | Create Pod modal opens; Select value resets to current pod (no navigation) |

### "Add Deck" Button (PlayerDecksTab — owner only)

| State | Behavior |
|-------|----------|
| Viewer is deck owner | "Add Deck" button renders (outlined, right-aligned, above DataGrid) |
| Viewer is not owner | Button hidden entirely |
| Clicked | Navigate to `/deck/new` |

## Mobile Contract

All new surfaces must satisfy the DSNG-02 mobile contract (375px viewport):

| Surface | Mobile requirement |
|---------|--------------------|
| HomeView empty state | Heading + body + CTA button centered; button full-width acceptable at xs |
| Create Pod modal | TextField full-width; Dialog `maxWidth="xs"` to prevent overflow |
| AppBar pod-switcher Select | Existing at 375px — no regression; "Create new pod" MenuItem must be reachable by scrolling within the dropdown |
| PlayersTab card layout | Cards stack vertically; full width at xs; icon buttons (Promote/Remove) render at minimum 44px touch target |
| /deck/new form | Container `maxWidth="sm"` centers on desktop; full-width at xs; all TextFields full-width; action buttons stack vertically at xs (Cancel above Save or below) |
| Player card stat row | Single line preferred; allow wrapping to two lines at 375px if stat values are long — body2 (14px) minimum |
